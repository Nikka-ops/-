# InterviewRadar 项目优化文档

## 1. 文档目标

本文档用于整理 InterviewRadar 当前的工程现状、主要问题、优先级排序，以及面向普通用户的低门槛优化方案。

目标不是讨论功能创意，而是明确三件事：

1. 这个项目现在处于什么阶段
2. 哪些问题最影响真实可用性
3. 应该按什么顺序优化，才能让普通用户尽可能低门槛运行

---

## 2. 当前项目判断

### 2.1 整体评价

项目整体处于 **中上水平的 Beta 阶段**。

它不是简单的脚本拼接，也不是纯演示型 Demo，已经具备比较清晰的产品雏形和工程分层：

- 有明确的产品目标：面经抓取、题库生成、岗位分析、模拟面试
- 有相对完整的模块拆分：`connectors / corpus / jobs / api / ocr / tools`
- 有 Web UI、CLI、测试、CI、安装脚本等基本工程配套
- 代码中能看出作者在考虑可维护性，而不只是“先跑起来”

但它距离“普通用户可顺畅使用的成品”还有明显差距。

### 2.2 当前更像什么

更准确地说，这个项目目前更像：

- 开发者自己可维护、可演示、可继续迭代的产品原型
- 有一定技术用户可用性，但需要踩坑和自行排障
- 不适合直接面向普通用户分发

### 2.3 当前是否能运行

结论：**有条件能跑，但不够稳，不够低门槛。**

面向开发者：

- 只要 Python 版本正确
- 依赖安装完整
- 配置了必要环境变量
- 理解外部抓取依赖和登录态要求

大概率可以跑起来。

面向普通用户：

- 安装链路复杂
- 外部依赖多
- 错误提示不够产品化
- 启动默认依赖不够友好

因此不能算“开箱即用”。

## 3. 主要问题清单

以下问题按优先级排序。

### P0：影响核心可用性的问题

#### 3.1 `--role-id` 没有正确透传到主流程

问题：

- CLI 中 `--role-id` 的输入没有完整写入 `RunConfig.role_id`
- 导致依赖 `role_id` 的查询扩展、角色预设、缓存元信息和后续角色聚合能力退化

影响：

- 功能表面可运行
- 但预设岗位能力并没有真正生效
- 属于行为级缺陷

结论：

- 必须优先修复

#### 3.2 安装兼容路径不完整

问题：

- `requirements.txt` 标注为兼容旧安装方式
- 但其中缺少运行时必需依赖，如 `pyyaml`、`rapidfuzz`

影响：

- 用户可能安装成功
- 但运行时直接报模块缺失
- 这是典型的发布链路缺陷

结论：

- 必须优先修复

#### 3.3 Python 版本兜底策略不严谨

问题：

- 项目要求 Python `>=3.11`
- `install.sh` 在找不到 `python3.11` 时回退到 `python3`
- 如果本机 `python3` 是 3.9/3.10，脚本仍会继续执行

影响：

- 用户会得到错误环境
- 报错位置后移
- 安装体验差，定位成本高

结论：

- 必须在安装前做强校验并中止

### P1：影响用户排障和项目可信度的问题

#### 3.4 错误文案存在字符串插值错误

问题：

- 某些错误信息中会直接显示 `{config.role}` 字面量，而不是实际岗位名

影响：

- 不影响主逻辑
- 但会削弱错误信息的可读性和可信度

结论：

- 应尽快修复

#### 3.5 仓库元数据不一致

问题：

- README 克隆地址与 `pyproject.toml` 中的仓库链接不一致

影响：

- 用户无法明确哪个仓库是主仓
- 影响 issue 回流、包信息展示和分发可信度

结论：

- 应尽快统一

### P2：影响长期交付质量的问题

#### 3.6 测试覆盖偏业务，不够覆盖发布链路

现状：

- 解析器、服务层、API、岗位抓取已有较多测试
- 这是优点

缺口：

- 没有覆盖安装脚本正确性
- 没有覆盖 Python 版本校验
- 没有覆盖 `role_id` 透传
- 没有覆盖“无 API Key 启动”这类真实用户场景

影响：

- 项目容易在“代码逻辑正确”但“用户无法启动”这类问题上失守

结论：

- 后续需要补齐交付型测试

## 4. 根本问题：项目还不是交付物

当前最大的问题不是某一个函数 bug，而是整个项目仍然以“开发者使用方式”为中心。

普通用户看到的是：

- 需要 Python 环境
- 需要依赖安装
- 需要环境变量
- 需要理解 DeepSeek
- 需要理解小红书/Boss 抓取前置条件
- 需要理解本地缓存和语料来源

这对普通用户是不成立的。

所以优化重点不应只是继续加功能，而应把项目从：

- 开发者项目

改造成：

- 面向最终用户的交付物

---

## 5. 面向普通用户的优化目标

普通用户低门槛运行的标准应当是：

1. 不需要手动创建虚拟环境
2. 不需要理解 Python 版本问题
3. 不需要先配置 API Key 才能看到结果
4. 不需要理解抓取依赖才能使用基本功能
5. 启动失败时能看到人话提示，而不是模块异常

换句话说，用户第一次接触项目时，目标不应该是“完成配置”，而应该是“先看到结果”。

---

## 6. 优化策略

### 6.1 先做“最小可用模式”

建议默认提供一个无需配置的基础模式：

- 启动后直接加载样例数据或内置语料
- 可查看面经列表
- 可查看题库
- 可体验模拟面试基础流程

这一步不要强依赖：

- DeepSeek API Key
- 小红书登录态
- Boss 登录态
- 实时抓取

原因：

- 普通用户第一次启动时，核心任务是确认“这个产品是什么”
- 而不是先花 20 分钟解决环境配置

### 6.2 把联网增强能力改成“可选”

建议将能力分层：

#### 基础模式

- 使用本地样例数据或最近缓存
- 不依赖 AI Key
- 不依赖实时抓取

#### 增强模式

- 配置 AI Key 后启用题目解答、语义聚类、JD 分析

#### 抓取模式

- 配置小红书/Boss 前置条件后再开启实时抓取

这样能避免用户一上来就因为外部系统而失败。

### 6.3 提供唯一官方启动方式

当前项目不应向普通用户暴露多种安装入口。

应只保留一个主路径，例如：

- macOS: `start.command`
- Windows: `start.bat`

脚本内部完成：

1. 检查 Python 版本
2. 创建虚拟环境
3. 安装依赖
4. 检查运行模式
5. 启动服务
6. 自动打开浏览器

原则：

- 用户只做一次双击或一条命令
- 其余细节由脚本处理

### 6.4 安装失败要尽早失败

安装脚本必须在前置阶段明确校验：

- Python 是否存在
- Python 版本是否 >= 3.11
- 依赖是否安装成功
- 启动端口是否可用

遇到问题时应直接输出明确提示，例如：

- “检测到 Python 3.9，不满足最低版本 3.11，请先安装 Python 3.11+”
- “未配置 AI Key，已自动切换为基础模式”

不要把错误留到导入阶段才暴露。

### 6.5 用设置页替代 `.env` 编辑

普通用户不应该被要求手工创建或编辑 `.env`。

建议新增设置页面，用于：

- 填写 API Key
- 检测连接
- 查看当前模式
- 查看数据源状态
- 启用/关闭抓取功能

配置应保存到用户目录，而不是要求用户直接接触工程文件。

### 6.6 抓取能力与主产品解耦

小红书/Boss 抓取属于高波动、高前置依赖能力，不应成为主体验的门槛。

建议：

- 默认首页先展示样例数据或缓存数据
- 抓取能力放入“高级功能”
- 页面明确标注抓取依赖和可能失败原因

目标是：

- 就算抓取失效，项目主体仍可使用

### 6.7 提供桌面打包版本

如果目标真的是普通用户，最终最有效的方案不是优化 README，而是直接打包。

建议优先探索：

- `PyInstaller`
- `Nuitka`

打包目标：

- 后端服务可执行程序
- 内置静态前端资源
- 双击即启动

这是降低门槛最有效的路径。

## 7. 建议的优化优先级

### 第一阶段：先修“能不能正确用”

目标：让技术用户稳定跑起来。

建议顺序：

1. 修复 `role_id` 透传问题
2. 修复 `requirements.txt` 缺失依赖问题
3. 修复安装脚本 Python 版本校验
4. 修复明显错误文案和仓库元数据不一致问题

产出标准：

- 开发者按 README 能更稳定跑通
- 核心功能行为与文档一致

### 第二阶段：先做“无配置可体验”

目标：让普通用户第一次启动能看到结果。

建议顺序：

1. 增加基础模式
2. 启动默认加载样例数据或缓存
3. AI Key 缺失时自动降级
4. 增加健康检查与人话提示

产出标准：

- 用户不配任何 Key 也能打开产品并体验主要界面

### 第三阶段：再做“一键交付”

目标：减少环境依赖暴露。

建议顺序：

1. 统一启动入口
2. 新增设置页
3. 把抓取能力移到高级功能
4. 增加桌面打包版本

产出标准：

- 用户不需要理解 Python、虚拟环境和 `.env`

### 第四阶段：补交付型测试

目标：防止后续改动破坏启动和分发体验。

建议新增测试：

1. `role_id` 透传测试
2. 安装脚本最小烟雾测试
3. Python 版本失败测试
4. 无 API Key 基础模式启动测试
5. 样例数据模式启动测试
6. 打包产物启动测试

## 8. 推荐的一周改造方案

如果按一周可落地的强约束来排，建议聚焦以下内容。

### Day 1-2：修基础缺陷

- 修 `role_id` 透传
- 补齐 `requirements.txt`
- 收紧 `install.sh` 版本校验
- 统一 README 和包元数据链接

### Day 3-4：做基础模式

- 启动时默认启用样例数据
- 无 AI Key 时自动降级
- Web UI 明确显示当前运行模式

### Day 5：补健康检查

- 启动前检查依赖、Python 版本、配置状态
- 错误提示改为可理解文案

### Day 6-7：统一启动入口

- 增加 `start.command` / `start.bat`
- 自动打开浏览器
- 尽量隐藏环境准备细节

如果时间更充足，再进入打包阶段。

---

## 9. 最终建议

这个项目值得继续做，但方向上要从“功能继续扩张”切换到“交付体验收口”。

当前最需要的不是再多一个抓取器或再多一个分析视图，而是：

1. 让现有功能行为与文档一致
2. 让安装和启动路径稳定
3. 让用户即使不配置外部能力也能先看到结果
4. 让抓取和 AI 能力变成增强项，而不是门槛

如果这四件事做好，项目会从“技术上有意思”明显提升到“真正能被更多人用起来”。
